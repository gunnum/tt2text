# App 分析报告模块

当前 App 分析模块按正式报告的阅读顺序整理如下。原「市场表现」不再单列为独立报告模块，已经并入模块 1「数据状态判断」：模块 1 负责先回答这个 App 为什么值得看，以及下载、收入、活跃、排名、国家结构等市场数据当前处在什么状态。模块 1 的正式标题不固定写模块名，而是由 Agnes 根据真实数据动态生成，例如「下载回落，收入保持韧性」「收入稳增，下载放缓」「排名高位稳定」。

| 模块 | 名称 | 主要回答什么 | 主要来源 | 自动化程度 | 当前代码状态 |
| --- | --- | --- | --- | --- | --- |
| 1 | 数据状态判断 / 动态数据标题 | 这个 App 为什么值得看？收入、下载、活跃、排名、国家结构当前是什么状态；正式标题根据实际数据灵活总结 | ST overview、ST CSV、榜单、国家市场数据、文章或 TikTok 声量补充 | 高 | 已接入 Agnes prompt；覆盖原「市场表现」核心问题 |
| 2 | 国家市场分工 | 哪些国家拉下载，哪些国家贡献收入，哪些市场付费强度高 | ST 国家下载/收入/RPD | 高 | 已接入 Agnes prompt |
| 3 | 垂类竞品榜 | 它在同品类里排第几，前后都是谁，哪些竞品未入库 | ST 垂类排行榜 | 高 | 已接入 Agnes prompt |
| 4 | 用户痛点 | 用户为什么需要它？它满足了什么痛点，省掉了什么成本 | 商店文案、评论、TikTok、文章、你输入的 MD | 中 | 已接入 Agnes prompt |
| 5 | 亲测体验 | 实际使用下来流程、体验、问题、关键判断是什么 | 你写好的体验 MD | 高，依赖输入 | 已有模块定义和质量产物生成，依赖体验 MD |
| 6 | 增长内容 / 传播表达 | 外部内容怎么包装它？hook、卖点、演示、评论反馈是什么 | TikTok 视频、转写、评论、素材 | 高 | 已接入 Agnes prompt |
| 7 | 付费点与价格结构 | 它卖什么、锁什么、哪些市场更愿意付费 | paywall、IAP、ST 国家 RPD | 中高 | 已接入 Agnes prompt |
| 8 | 用户评论与口碑 | 用户喜欢什么、抱怨什么、有哪些风险信号 | ST reviews、App Store 评论、TikTok 评论 | 高 | 已接入 Agnes prompt |
| 9 | 创始人 / 公司 / 融资背景 | 公司和创始人有什么背景，是否有融资、PR、收购、转型线索 | 已录入文章、PR 稿、访谈 | 条件稳定 | 已接入 Agnes prompt；依赖 A 类文章证据 |

## 当前 Prompt 状态

- 已确认：模块 1「数据状态判断 / 动态数据标题」prompt；模块 2「国家市场分工」prompt；模块 3「垂类竞品榜」prompt；模块 4「用户痛点」prompt；模块 6「增长内容 / 传播表达」prompt；模块 7「付费点与价格结构」prompt；模块 8「用户评论与口碑」prompt；模块 9「创始人 / 公司 / 融资背景」prompt。
- 引用规则：Agnes prompt 已要求正文少写“某媒体报道称 / 根据某采访”，事实句后保留 A/D/K/R/C/V/P/S/X 等证据 ID；前端阅读版会把 A 类文章证据渲染为可点击引用和 hover 预览卡。
- 已并入：原「市场表现」的整体规模、趋势、收入、下载、活跃、排名等判断并入模块 1。
- 待确认：模块 5「亲测体验」依赖手写体验 MD，暂不接 Agnes prompt。
- 代码注意：`server/report-output-service.mjs` 里仍保留 `market_overview` 这个历史模块定义，用于旧产物和确定性质量框架；正式模块口径以本文档的 9 个连续编号为准。后续代码层可考虑把 `market_overview` 彻底迁移进 `anomaly_signal`，避免 UI 或批量生成继续暴露重复模块。

## 模块边界与重叠检查

| 可能重叠 | 边界建议 |
| --- | --- |
| 模块 1 数据状态判断 vs 模块 2 国家市场分工 | 模块 1 只写整体数据状态和最关键的国家结构信号；模块 2 专门展开国家之间的下载、收入、RPD 分工和错配，不重复整体规模。 |
| 模块 1 数据状态判断 vs 模块 3 垂类竞品榜 | 模块 1 可引用榜单说明当前位置；模块 3 只做同品类排名、前后竞品、未入库竞品和补采入口，不写整体增长趋势。 |
| 模块 4 用户痛点 vs 模块 6 增长内容 | 模块 4 回答用户为什么需要这个产品；模块 6 回答外部内容如何表达和放大这个需求。素材里的 hook 可以进入模块 6，只有能证明用户任务或痛点时才进入模块 4。 |
| 模块 4 用户痛点 vs 模块 8 用户评论与口碑 | 模块 4 从评论中提炼需求和成本节省；模块 8 保留评论原话、好评/差评主题和风险信号，不把所有评论都改写成痛点。 |
| 模块 5 亲测体验 vs 模块 8 用户评论与口碑 | 模块 5 以实测流程为主，评论只做验证或反证；模块 8 以用户样本为主，不替代亲测结论。 |
| 模块 7 付费点与价格结构 vs 模块 2 国家市场分工 | 模块 2 只说哪些国家 RPD 或收入占比高；模块 7 才解释 paywall、IAP、权益边界和高付费市场之间的关系，不推 LTV、CAC 或续费率。 |
| 模块 7 付费点与价格结构 vs 模块 8 用户评论与口碑 | 模块 7 写产品卖什么、锁什么；模块 8 写用户对价格、订阅、广告或退款的评价与风险。 |
| 模块 9 创始人 / 公司 / 融资背景 vs 其他模块 | 模块 9 只引用已录入文章、PR 或访谈中的事实；只有当公司背景能解释产品路线、增长动作或商业化选择时，才在其他模块中作为补充。 |

## 当前 Workflow

- 数据预清洗和 Agnes 调用统一走 `docs/report-ai-workflow.md`。
- 录入或更新 Sensor Tower 数据后，先用 `node scripts/run_report_ai_module.mjs --app calai --module country_market_split --mode prepare` 生成 source pack 和 prompt。
- 验证 prompt 时再跑 `--mode generate`，避免每次临时拼数据。
